# Internationalization ŌĆö next-intl (Cookie-Based)

## Architecture

This app uses **cookie-based locale resolution** ŌĆö there is no `[locale]` URL segment and no localized routing. Switching language rewrites a cookie and refreshes the current route.

- **Locale registry** (`i18n/routing.ts`): the single source of truth ŌĆö `locales` as a `const` tuple, the `AppLocale` type derived from it, `defaultLocale`, the locale cookie name, and the guards (`isAppLocale`, `resolveAppLocale`). Every other file imports from here; never re-type locale literals elsewhere.
- **Request config** (`i18n/request.ts`): `getRequestConfig` resolves the locale per request ŌĆö cookie first, then the `Accept-Language` header, then the default ŌĆö and loads that locale's messages JSON.
- **Proxy/middleware**: seeds the locale cookie on first visit (from `Accept-Language`) so subsequent requests resolve consistently.
- **Root layout**: wraps the app in `NextIntlClientProvider`; the next-intl plugin is registered in `next.config.ts`.

---

## Message Files

- One JSON file per locale under `i18n/locales/` (e.g. `en.json`, `tl.json`).
- Messages are organized into **namespaces per surface or feature** (top-level keys), consumed as `useTranslations('Namespace')`.
- **All locale files must stay key-identical.** Adding a key means adding it to every locale file in the same change ŌĆö a missing key in one locale is a bug, not a follow-up.
- Use ICU parameters for interpolation (`{name}`, plurals) ŌĆö never build translated strings by concatenation.

## Consuming Translations

```tsx
// Client Components
const t = useTranslations('Common');
const locale = useLocale();
<Button aria-label={t('language')} />

// Server Components / metadata
const t = await getTranslations('Common');
```

- The locale always comes from the next-intl provider (`useLocale`) or request config ŌĆö **never read the locale cookie directly in client render** (hydration-unstable).
- Translate `aria-label`s and other assistive text, not just visible copy.
- Locale-aware date/number output derives its locale from `useLocale` (or next-intl formatters) ŌĆö do not hardcode a locale string at call sites.

## Language Switcher Pattern

Switching locale = persist the cookie, then refresh so Server Components re-render with the new request config:

```tsx
const [isPending, startTransition] = useTransition();

function selectLocale(nextLocale: AppLocale) {
  document.cookie = `${localeCookieName}=${nextLocale}; Path=/; Max-Age=31536000; SameSite=Lax`;
  startTransition(() => router.refresh());
}
```

- Disable the switcher trigger while the transition is pending.
- Import the cookie name and `locales` from the registry ŌĆö the switcher must not carry its own copy of either.

## Rules

- **Translated surfaces stay translated.** When editing a component that already uses `useTranslations`, new user-facing strings go into the message files as keys ŌĆö never hardcoded text next to translated text.
- **New keys land in every locale file in the same change**, in the same namespace position, so the files stay mirrored.
- **One locale registry.** Locale list, default, cookie name, and guards live in one module; everything imports from it.
- **No hydration-unstable locale reads.** Client components get the locale from the provider, not from `document.cookie` or `navigator.language` during render.
- Keep namespaces aligned with surfaces/features so unused namespaces are easy to spot and messages are easy to find.

---

## Anti-Patterns

- Hardcoding a user-facing string inside a surface whose siblings are translated.
- Adding a key to one locale file and deferring the others.
- Re-typing `'en'` / locale literals instead of importing from the registry.
- Concatenating translated fragments instead of using ICU parameters.
- Reading the locale cookie or `navigator.language` in client render.
- Introducing `[locale]` route segments or per-locale paths ŌĆö this app's model is cookie + refresh.
